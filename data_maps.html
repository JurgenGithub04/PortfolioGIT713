Database Management and Data Wrangling

Database Management

A database is a structured repository of data for easy access, modification and analysis, while a spatial database is designed to store and manage spatial data with a geographic or locational component. Database design involves several key steps such as requirements gathering, conceptual design, logical design, normalization, physical design, implementation, and testing. Spatial database design requires specialized techniques and considerations, such as selecting an appropriate data model, choosing spatial indexing techniques, and considering coordinate systems and spatial data transformations. The database design process starts with data abstraction, which progresses from a high-level overview to more detailed levels of refinement, including a conceptual model, logical model, and physical model.

Database Management for Botanical Garden Project

Managing the database for our Botanical Garden project was a crucial task, as it allowed us to retrieve essential information for creating a web map that met the client's requirements. Before starting the project, I made sure to review the geodatabase carefully and identify the features that we needed to retrieve, such as the garden boundaries and plant bed details. By doing this, I was able to extract only the relevant information from the database, which helped streamline the rest of the process.

Another critical aspect of the project was the species data. We received the data in an Excel file, and I had to create mock data for the species themes, which was an interesting challenge. I had to ensure that the data was structured in such a way that it allowed plants to have multiple themes. This was essential because it would enable the client to add more themes to the database later on, making it more flexible for future updates. Additionally, I made sure to clean the data by removing any null values and inaccurate coordinates. This ensured that the data was reliable and consistent, which was necessary for creating an accurate web map.

Once the data was cleaned and structured correctly, I converted it to GeoJSON format. This was a new experience for me, and I learned a lot about the structure of GeoJSON data and how it worked. It was crucial to understand the format to ensure that the data was compatible with the Leaflet map we were using.

Working in a team was an essential part of this project. Charles, one of my group members, helped us by writing the code to convert the features to GeoJSON. This was a massive help, as it saved us a lot of time and effort. Collaborating with others allowed us to combine our strengths and improve the overall quality of the project.

Finally, producing a web map that met the client's requirements effectively was a satisfying experience. The project taught me valuable skills in database management, data cleaning, and conversion to GeoJSON format. It also showed me the importance of working with others and how collaborating can enhance the final product's quality. Overall, it was a challenging but rewarding experience, and I learned a lot from it.
